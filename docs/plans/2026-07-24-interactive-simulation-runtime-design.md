# 互动仿真播放可靠性升级

## 问题

互动页本身和课程播放时间轴分别生成。当前时间轴把学生操作等待点放在大部分
`widget_*` 自动动作之后，造成讲稿要求学生操作、系统却先替学生改变状态。页面
完成信号如果未被触发，播放又只能依赖较长的超时，用户感知为“卡住”。

## 设计

1. **学生先操作**：互动场景的顺序固定为“操作提示 → 可选控件聚焦 → 学生活动门
   → 反馈/演示”。`widget_setState`、`widget_reveal` 和 `widget_annotation` 不得在
   活动门前替学生完成任务。
2. **新旧场景同一规则**：生成时按该顺序落盘；播放引擎加载场景时也做一次无损
   规范化，使当前课堂立即生效，不要求重新生成课程。
3. **事件优先、超时兜底**：页面通过 `window.__maicActivity.complete()` 或
   `data-activity-complete` 提交学习证据后立即继续；等待时间限制在合理区间，避免
   异常配置造成无限等待或几秒后自动跳过。
4. **可见且可恢复**：等待期间显示“正在等你完成页面操作”的状态，并提供
   “我已完成，继续讲解”人工兜底。人工按钮不是正常流程的“下一步”，只用于页面
   脚本未发出完成事件时恢复课程。
5. **生成契约**：互动动作提示明确禁止用平台动作代替讲稿要求的学生点击、选择、
   拖拽或参数调整；平台动作只负责聚焦和完成后的解释性演示。

## 验证

- 单元测试覆盖新生成与旧时间轴的活动门位置、等待时长边界和事件提前完成。
- iframe 协议测试继续保证完成/重置标记被注入。
- 运行相关测试、TypeScript/生产构建，确认播放器和生成流程没有回归。
