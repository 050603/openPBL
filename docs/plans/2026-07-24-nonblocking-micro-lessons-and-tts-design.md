# 非阻塞微课与课堂 TTS 修复设计

## 目标

学生在 proposal / make 阶段提出系统讲解需求后，不再被全屏生成浮层阻塞。微课在后台生成，伴学课堂保持可用；知知进入工作状态，界面边角任务卡持续展示进度。生成完成后由任务卡和知知语音共同提醒，学生主动进入独立学习页，完成后返回原伴学课堂。

同时修复主 AI 授知课堂和即时微课无声的问题。两类课堂继续复用同一套场景、播放器和 TTS 管线。

## 交互与状态

微课任务使用 `generating → ready → completed / failed` 状态。伴学运行时只等待“是否需要系统讲解”的轻量判断；确认后立即恢复输入能力，并启动独立后台生成任务。任务进度进入运行时上下文，由工作室任务卡、顶部状态和 Pixi 智能体状态共同消费。

生成中，任务卡显示主题、生成消息和百分比，知知使用现有 `working` 动画。完成后卡片切换为可操作状态，显示“进入学习”；失败时保留错误提示并允许关闭。独立微课页面使用 `StudentStageHost` 的 standalone 模式，避免写入主 AI 授知进度。所有场景完成后显示完成状态，返回按钮将微课标记为完成并回到原课堂；提前返回则保留待学习状态。

## TTS 修复

播放器优先播放已生成音频。当新课堂的服务端音频尚未生成或加载失败时，只要学生课堂已显式启用 browser-native-tts，就立即使用浏览器语音兜底，不再要求它同时是当前选中的服务端提供商。学生播放器每次加载时恢复非静音和有效音量，避免历史设置导致整堂无声。

服务端媒体生成继续限制教师资源，但把 `audience=student`、`generationPurpose=knowledge-teaching` 且位于 proposal / make 的即时微课纳入 TTS 范围。

## 验证

使用单元测试覆盖浏览器 TTS 兜底条件和服务端微课场景筛选；再执行相关功能测试、TypeScript 检查与生产构建。界面验证重点包括：生成期间可继续发送请求、任务卡不遮挡输入区、知知工作动画、完成提醒、独立学习页和返回闭环。
