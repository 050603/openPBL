# 课程资源分流与 TTS 修复计划

> 目标：修复 PBL 后续阶段资源误进入学生 AI 授知列表并被 TTS 配音的问题，同时将教师资源并入“普通课堂活动”的标签语义。

## 根因假设

1. 生成 API 在 `generateClassroom` 返回后，先把包含全部场景的原始课堂关联到课程；浏览器端的后置拆分随后才执行。因此在拆分未完成、失败或页面离开时，课程仍可能指向全量课堂。
2. 当前 TTS 在拆分前对全量场景执行，教师/普通课堂场景一旦泄漏到学生端，就已经带有音频。
3. 场景分类虽然优先读取 `audience`、`stageKey` 和 `generationPurpose`，但需要确认 Stage API 创建场景时完整保留这些字段；若落库丢失，分类会退化为标题关键词判断。
4. 前台仍把 `teacher-resource` 当作独立标签，导致“普通课堂活动”与教师支持资源的生成语义分裂。

## 实施方案

### 1. 验证元数据与链路

- 检查 Stage API 的场景创建与存储类型，补充 PBL 路由元数据的持久化测试。
- 检查生成 API、课程关联和浏览器端后置拆分的时序，确保学生课堂只在服务端完成分流后关联。

### 2. 服务端先分流再关联

- 抽取可复用的服务端课堂拆分函数：依据显式 PBL 元数据生成学生课堂和普通课堂教师支持课堂。
- 生成 API 在课程关联前执行拆分；课程仅关联学生课堂，同时保存教师课堂及资源快照信息。
- 生成完成事件返回拆分统计与教师资源信息，前端只同步结果，不再负责关键隔离动作。

### 3. 固化 TTS 与资源类型规则

- TTS 只对 `audience=student`、`stageKey=ai-learning`、`generationPurpose=knowledge-teaching` 的场景执行。
- 普通课堂活动的教师支持内容只生成 PPT/讲稿，不生成 TTS；学生 AI 授知仍可生成互动内容和 TTS。

### 4. 收敛标签语义

- 前台只展示“学生 AI 授知”和“普通课堂活动”两个标签。
- 将历史/模型输出中的 `teacher-resource` 归一化为普通课堂活动；内部生成用途元数据仍可用于技术分流，但不再作为教师可选标签。
- 更新校验、预览、生成摘要与提示文案。

### 5. 验证

- 增加服务端分流、元数据保留、TTS 场景选择和标签归一化测试。
- 执行单元测试、类型检查、生产构建、Lint/差异检查，并核对生成链路中没有重新关联原始全量课堂的路径。
