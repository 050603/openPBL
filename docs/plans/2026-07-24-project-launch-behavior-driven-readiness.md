# 项目启动阶段行为驱动待办设计

## 目标

把学生端“点击待办、再次确认、提交完成”的通用流程，改成由真实学习行为驱动的准备状态，同时修复学生身份写入待办与伴学确认时的权限错误。

## 交互模型

- 浏览课程资源：教师未上传资源时自动完成；存在资源时，学生逐一打开浏览，全部浏览后自动完成。
- 选择研究主题：从课程知识点和学习目标生成可选主题，允许学生自定义；保存到个人项目空间后自动完成。
- 确认个人项目空间：系统已自动创建，不再向学生展示。旧课程中的同名待办会自动完成。
- 其他教师自定义待办：保留简洁的直接完成入口，不再经过伴学二次确认。

## 权限模型

- 新增 `SET_STUDENT_TODO_COMPLETION`，学生只能提交自己的 `studentId`，服务端只增删该学生的完成记录。
- `UPSERT_TODO` 继续保持教师权限，避免学生覆盖整条待办或其他学生的状态。
- 学生更新研究主题时携带本人 `studentId`，reducer 只允许项目组成员更新对应项目。
- 伴学确认的创建与处理允许学生本人执行，并校验嵌套确认记录和处理请求中的学生身份。

## 兼容策略

不迁移旧课程数据。界面根据旧待办 ID 和标题识别三种默认待办，因此旧课程进入启动页后也能获得新的行为驱动体验；新课程只创建“阅读项目说明”和“选择研究主题”两项。
